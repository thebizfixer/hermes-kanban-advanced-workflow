# test-plan — Kanban Advanced Example Test Plan & Artifacts

This folder contains a complete example plan for the `kanban-advanced` Hermes Agent plugin. Use it to calibrate your plugin installation and verify that the governance pipeline (preflight → attest → decompose → evaluation chain → audit → reconcile → postmortem) works end-to-end **in your target repository** before you run your own production plans.

## Quick Start

```bash
# 1. Copy the plan to your target repo
cp test-plan/kanban-standard-smoke-test.plan.md /path/to/your-project/.hermes/kanban/plans/

# 2. From your target repo, run the gates
cd /path/to/your-project
bash "$HERMES_HOME/scripts/preflight.sh"
bash "$HERMES_HOME/scripts/pre_dispatch_gate.sh" kanban-standard-smoke-test
python3 "$HERMES_HOME/scripts/kanban_handoff.py" \
  --plan .hermes/kanban/plans/kanban-standard-smoke-test.plan.md
```

The plan file is the **single source of truth**. Once copied to your target repo, it's yours — the plugin repo's copy is just a template.

## Contents

| File | Purpose |
|------|---------|
| `kanban-standard-smoke-test.plan.md` | A standardized smoke test plan (5 cards: create utilities, test them, modify, negative governance test, verification) that exercises every phase of the kanban-advanced SOP. |
| `scripts/` | **Generated by the coding agent** — populated at runtime in the target repo (not here). Gitignored in the target repo. |
| `artifacts/` | **Generated at runtime** — post-run artifacts in the target repo (postmortem, KPI JSON, reconciliation reports). Gitignored in the target repo. |

> **Note:** `smoke_utils.py` and `test_smoke_utils.py` are **not shipped** — they are created by the coding agent when you run the smoke test plan in your target repo. The plan file is the single source of truth. This keeps repos clean of generated code and lets users verify that their coding agent produces correct output.

**Key principle:** Scripts live in `$HERMES_HOME/scripts/` (materialized by `hermes kanban-advanced init`). The plan lives in the target repo. All generated artifacts stay in the target repo. Nothing in the plugin clone is needed at runtime.

## How to Use

1. **Install the plugin** if you haven't already:
   ```bash
   hermes plugins install thebizfixer/hermes-kanban-advanced-workflow
   ```

2. **Initialize** your target project (replace `<your-project>` and `<branch>`):
   ```bash
   cd <your-project>
   hermes kanban-advanced init --project-root . --working-branch <branch>
   ```
   This materializes all governance scripts to `$HERMES_HOME/scripts/`.

3. **Copy the test plan** into your project's kanban plans directory:
   ```bash
   cp test-plan/kanban-standard-smoke-test.plan.md <your-project>/.hermes/kanban/plans/
   ```
   (The plugin repo's `test-plan/` directory is a template. You only need the `.plan.md` file.)

4. **Run the full SOP** — from your target repo root (see [README.md](../README.md) for the flowchart):
   - Run preflight, pre-dispatch gate, and handoff from the target repo
   - The coding agent will generate `test-plan/scripts/smoke_utils.py` and `test-plan/scripts/test_smoke_utils.py` during execution. Do not create these files manually — the plan's Cards 1–3 handle creation and modification.

5. **Verify your results** match the expected output:
   - 8/8 tests pass (greet, add, format_name, multiply)
   - Evaluation chain gates exercised (E002 governance, E018 token exactness, E023 error attractors)
   - Escalation ladder demonstrated (board-keeper 5-loop cap → tracker → orchestrator)
   - Postmortem generated with token economics + KPI JSON
   - Waste ratio reported (target <20%)

## What Good Output Looks Like

After a clean run in your target repo, you should see:
- **Board**: 5 work cards + gate + decompose + audit, all completed
- **Tests**: `python3 -m pytest test-plan/scripts/test_smoke_utils.py -v --rootdir=test-plan/scripts` → 8 passed
- **Postmortem**: `<plan_id>_postmortem_<date>.md` with execution summary, token economics, agent performance, pitfalls
- **KPI JSON**: `<plan_id>_kpi.json` with success rate, intervention rate, thrash outliers, blocker chains
- **Reconciliation**: `<plan_id>_reconciliation_<date>.md` with thrash analysis, parser health, audit health
